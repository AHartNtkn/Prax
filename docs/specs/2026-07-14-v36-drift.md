# v36 — Decay & drift (`Prax.Drift`): episodic state on the clock

User-directed, with the scope sharpened in review before any code:

- **Episodic state only.** Time changes appetites, intoxication, arousal — never
  dispositions. Grudges, conduct marks, trust, standing change through ACTS (confession,
  absolution, amends — the machinery already built), and time-decay on any of them is
  REJECTED: it would undermine atonement-not-amnesia (discharge must cost something) and
  history-persists-through-marks (truth becoming unrecoverable is drama, not bookkeeping).
  Games represent hours-to-weeks; nothing dispositional cools in that window.
- **Stamps never change.** Freshness (`atSince` + `sightedWithin`) is a static stamp
  compared against the moving clock at query time — a different mechanism, already built,
  untouched here. The declined recency-gradient stays declined (LEDGER).
- **Rates are authored world parameters with stated semantics** — a period means something
  in the fiction ("a mealtime", "how fast drink wears off"), never a tuned constant.
- The mechanism test that scopes cargo: **does the ticker have to MUTATE something?**
  Yes ⇒ drift. No ⇒ not this round.

## Design: pulse rules on a bodiless drifter (zero engine surface)

`Prax.Drift` is the v18/`Prax.Sight` idiom pointed at state evolution — a compiler of
authored rules into ordinary practice content; no engine, planner, or query change:

```haskell
data DriftRule = DriftRule
  { driftRuleName :: String        -- single segment; loud error otherwise
  , driftPeriod   :: Int           -- rounds between pulses; authored meaning stated
  , driftBody     :: [([Condition], [Outcome])]  -- ForEach clauses applied per pulse
  }

driftName  :: String               -- "_drift" (bodiless, blank label, practice-bound)
driftChar  :: Character
driftP     :: [DriftRule] -> Practice
driftSetup :: [DriftRule] -> [Outcome]  -- instance + per-rule due.<name>!0 seeds
```

Compiled shape, per world: ONE blank-label action on the drifter whose outcomes carry,
per rule, a due-gated bundle — `ForEach ([ Match "due.<name>!D", Match "turn!Now",
Cmp Gte "Now" "D", Calc "D2" Add "Now" "<period>" ] ++ clauseConds) (clauseOuts)` for each
body clause, plus the due bump `ForEach [due-gate…] [ Insert "due.<name>!D2" ]`. The due
fact makes the pulse cadence exclusion-logic-native (no modulo, no new arithmetic): a rule
fires when the clock reaches its due, then re-arms `period` rounds ahead. Multiple rules
ride one action independently (a ForEach whose gate fails simply does not fire).

The drifter depends on the sight clock (`turn!N`). A drift practice in a world without the
clock would silently never fire — banned; `prax check` gains a well-formedness rule: a
world registering the drift practice without `turn!0` (or without the sight ticker) is
flagged loudly.

**v35 compatibility is a design rule, not an accident**: drift effects must exist only at
authored thresholds — a pulse either inserts/clears a gate-shaped fact (`hungry.<who>`) or
moves a counter whose sole consumers are threshold checks (`drinks!N` → tipsy at ≥2). No
want may read raw sub-threshold ticking; then a pulse wakes exactly the characters whose
gates or satisfaction it actually moved (the motive signature does the rest), and intention
persistence survives the clock. The round re-measures the paired drive bench with drift
active and reports the serve-rate erosion as measured.

## Cargo (one shape, two worlds, both with story stakes)

1. **Village — the hunger pulse (build-up).** An `appetite.<who>` fact family marks
   bearers (bob first). Every `mealPeriod` rounds: `ForEach [appetite.X, Not "hungry.X"]
   [Insert "hungry.X"]` — everyone hungers on the same pulse (mealtimes; the stated
   semantics). A vocabulary desire `suffers-hunger` (`Want [Match "hungry.Owner"] <0` —
   negative, so v33 FloorCheck keeps the sated cheap) plus an `eat` affordance (consumes a
   held loaf, deletes the hunger) close the loop: bob's bread economy becomes CYCLIC —
   earn, eat, hunger again — instead of a one-shot want that goes dormant after the first
   loaf. Free play will change; the village golden is re-captured in its own commit with
   the drift itemized line by line.
2. **Bar — metabolism (wear-off).** Every `soberPeriod` rounds, each patron's
   `practice.patron.P.drinks!N` decrements toward 0, and a `checkSober` mirror of
   `checkTipsy` clears `person.P.tipsy` when the count falls under the threshold. Drink
   enough fast enough and you stay tipsy; stop, and it wears off. Bar golden re-captured,
   itemized.

## Verification

- DriftSpec pins the compiler: a rule fires exactly AT its due, not before; the due
  re-arms period-ahead; two rules with different periods fire independently; a body
  ForEach covers every satisfying binding; the reserved-variable guards (`D`/`D2`/`Now`)
  reject colliding rule bodies loudly (the v30/v31 guard class); `prax check` flags a
  drift world with no clock.
- Village pins the cycle (hunger appears on the pulse; eating clears it; it returns next
  pulse; the golden's itemized drift shows bob's earn–eat rhythm) and the v33 interaction
  (`suffers-hunger` dead-now while sated — the pair-skip fires between meals).
- Bar pins the wear-off (tipsy at 2 drinks; sober after enough dry pulses; a topped-up
  patron stays tipsy) against the existing tipsy pins unmodified.
- ViewInvariant green; suite green throughout; goldens re-captured deliberately (world
  change, never engine change), each in its own commit, drift itemized.
- The paired drive bench re-run with the hunger pulse active: v35 serve-rate and drive
  time with/without drift, reported as measured.

## Out of scope

Threat expiry (mechanism deliberately undecided: stamp-window vs conversion vs belief
revision — decide when built); calendar & gatherings (the natural next round: recurring
clock-gated scene spawns); per-binding due stamps (no cargo needs them; add when one
does); hunger levels/starvation (binary hunger until a world needs more); any drift on
dispositions, ever (see the principle above).
