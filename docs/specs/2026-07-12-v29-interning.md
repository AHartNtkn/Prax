# v29 — Segment interning (designated by the v28 profile; exactness contract carried over)

Post-v28, the top cost centres are pure String work on *segments*: `mayUnifyNames`
(classification, ~25%) and `unifyNames` (String-keyed `Map` descent, ~25%) — over half the
round is String equality/ordering on short names that recur millions of times. Interning
replaces segment Strings with machine integers everywhere the engine computes; Strings remain
the authoring, display, golden, and serialization surface.

## The rule (unchanged)

Exact only: goldens byte-identical; ViewInvariant net green; identical `readView` contents
(compared, as today, at the String boundary). All nets compare *rendered strings*, so they are
representation-independent by construction — exactly what makes this change safe to attempt.

## Design

1. **`Prax.Sym`** (new): `newtype Sym = Sym Int` with a process-global intern pool
   (`unsafePerformIO`+`IORef`, `NOINLINE`, atomic updates — the canonical FastString-style
   technique; observably pure: within a run, `intern` is a function, and no symbol identity
   ever escapes to observable output, which is always rendered through `symName`).
   - **Variable-ness is packed into the id**: variables intern to odd ids, constants to even
     (decided once, at intern time, by the existing uppercase rule) — `symIsVar` is a bit
     test, no lookup, and the hottest predicate in unification becomes free.
   - `intern :: String -> Sym`, `symName :: Sym -> String`. Symbol ids are first-encounter
     ordered — run-dependent, which is why no id may ever be serialized, compared across
     processes, or rendered into output (haddock'd as the module's contract; `Persist` stays
     entirely String-based, so save files are unaffected).
   - Rejected alternative, recorded: a state-threaded symbol table (no global state) —
     runtime-fresh segments (numeric `Calc`/score/turn values) would force table-threading
     through `performOutcome`, `Db`'s public API, and every test fixture; the global pool
     confines the change to representation. Revisit only if multi-world symbol pressure ever
     matters (it cannot affect correctness, only pool size).
2. **`Db` re-keys on `Sym`**: `data Db = Db Bool (IntMap Db)`; tokens become
   `[(Sym, Maybe Char)]`; `unifySyms` descends by `IntMap` lookup with the bit-test variable
   check. The String API (`insert`, `unify`, `exists`, `dbToSentences`,
   `dbToLabeledSentences`, …) keeps its signatures — interning at entry, rendering at exit —
   so tests, worlds, Persist, and Inspect are untouched.
3. **`Val` becomes symbolic**: `VStr String` → `VSym Sym` (renamed — no dual constructors);
   `VSet [[String]]` → `VSet [[Sym]]`; `VNum` unchanged. `valToString` renders. `Bindings`
   keys become `Sym`. The handful of test fixtures constructing `Val` directly update.
4. **The cooked pipeline carries `Sym`**: `CookedCondition`/`CookedOutcome`/cooked containers
   swap `[String]`→`[Sym]` (cooking already happens once per world — interning rides it);
   footprints/negFootprint/shadows likewise; `mayUnifySyms` is Int equality + the bit test.
5. **`EL.meet`/`leq`** re-key mechanically (`IntMap` merges).

## Verification

- Goldens + ViewInvariant net + full suite green after every task (all compare Strings —
  representation-independent).
- `Prax.Sym` unit tests: intern/symName round-trip, variable parity packing, pool growth
  determinism within a run.
- Db equivalence pins: the String API's observable behavior (insert/unify/retract/sentences)
  asserted equal to pre-change expectations already encoded in DbSpec — DbSpec itself should
  need no semantic edits; needing one is a red flag to investigate, not accommodate.
- Honest perf report: profiled round + suite before/after; expected win is concentrated in
  `mayUnify*`/`unify*`/`insertToks` (~50%+ of the post-v28 round).

## Out of scope

- Hash-consing whole paths/tries, arena allocation, undo-log search (the Rust-port levers —
  recorded, not attempted here); approximate anything; new mechanics (standing directive).
